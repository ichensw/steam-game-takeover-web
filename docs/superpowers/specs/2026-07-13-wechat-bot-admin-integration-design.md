# 微信机器人后台统一接入设计

## 目标

将 `wechat-bot-admin-web` 的管理能力迁移到 `steam-game-takeover-web`，并以兔兔窝接龙后台管理员账号作为唯一身份来源。迁移后，管理员只登录一次即可按角色权限访问接龙和微信机器人管理功能。

## 涉及仓库

- `steam-game-takeover-web`：承载迁移后的微信机器人管理页面。
- `steam-game-takeover-backend`：作为统一身份权威和微信机器人接口网关。
- `wechat-bot-admin-backend`：继续提供机器人数据能力，只接受可信网关请求。
- `wechat-bot-admin-web`：发布停用通知，切换完成后停止部署。

## 架构选择

采用兔兔窝后端统一网关方案。浏览器只向兔兔窝后端发送现有 Bearer Token；兔兔窝后端验证管理员、会话状态和菜单权限后，将白名单内的请求代理到机器人后端。

不让机器人后端独立校验共享 JWT。兔兔窝管理员 token 还受数据库中的 token 撤销记录和账号启用状态约束，仅共享签名密钥会绕过这些检查。也不让机器人后端逐次回调鉴权，因为统一网关能避免跨域、双 API 地址和重复鉴权调用。

## 前端设计

在 `steam-game-takeover-web` 中新增“微信机器人”菜单组和三个原生 React/Ant Design 页面：

- `wechat-messages`：群聊消息查询、筛选和分页。
- `wechat-summary`：按群聊和时间范围生成 AI 总结，并导出总结图片。
- `wechat-database`：查看机器人数据库表、字段和分页数据。

页面沿用现有后台布局、加载状态、错误提示和表格规范。机器人账号管理不迁移；管理员统一在现有“管理员账号”页面维护。

前端 API 请求统一走兔兔窝后台的 `/api/admin/wechat-bot/*` 路径，并自动携带当前管理员 Bearer Token。前端菜单显示由当前管理员的 `menuKeys` 决定。

## 权限设计

新增三个菜单权限：`wechat-messages`、`wechat-summary`、`wechat-database`。

- 超级管理员始终拥有全部三个权限。
- Kook 管理员和普通管理员通过现有“角色菜单权限”页面配置。
- 菜单组在至少一个子权限存在时显示。
- 后端按请求路径和 HTTP 方法再次校验权限，不能依赖前端隐藏菜单。
- 群聊列表是消息查询和 AI 总结的共同依赖，对拥有任一相关权限的管理员开放。

## 网关设计

`steam-game-takeover-backend` 新增 `/api/admin/wechat-bot/*` 路由，放在现有 `AdminAuth` 之后。网关执行以下步骤：

1. 验证兔兔窝管理员 token、数据库会话记录和账号启用状态。
2. 根据请求方法和路径映射所需菜单权限。
3. 拒绝不在白名单内的路径和方法。
4. 删除客户端可能伪造的内部身份头。
5. 使用服务间共享密钥和已验证的管理员 ID、用户名调用机器人后端。
6. 透传机器人后端的 JSON 状态码和响应，并将连接失败、超时转换为统一网关错误。

服务地址、共享密钥和代理超时通过环境变量配置。AI 总结使用独立的较长超时，普通查询保持较短超时。

## 机器人后端鉴权

`wechat-bot-admin-backend` 的数据接口改用可信网关中间件：

- 使用常量时间比较校验服务间共享密钥。
- 只信任通过密钥校验后携带的管理员身份头。
- 未携带或密钥错误时返回 401。
- 停止注册旧 `/api/auth/*` 和 `/api/accounts*` 路由。
- 旧账号表和实现暂时保留但不再可达，避免在本次迁移中执行破坏性数据删除。

生产部署应优先让机器人后端仅在私有网络对兔兔窝后端开放。

## 停用旧前端

更新 `wechat-bot-admin-web/README.md`，在文件顶部发布明确的停用通知，包括：

- 项目已停止维护和部署。
- 功能已迁移到兔兔窝接龙后台管理系统。
- 管理员账号以兔兔窝后台为准。
- 仓库暂时保留用于历史追溯，不再接受新功能开发。

## 错误处理

- 无兔兔窝登录态：沿用现有 401 自动退出流程。
- 缺少菜单权限：返回 403，前端显示无权限提示。
- 网关配置缺失：服务启动或请求时给出明确配置错误，不回退到旧鉴权。
- 机器人后端超时或不可用：网关返回 502 或 504，前端保留当前筛选条件并提示重试。
- AI 总结失败：保留上一次结果，不生成空图片。

## 测试与验收

后端测试覆盖：

- 有效、撤销、过期 token 和禁用管理员。
- 超级管理员默认权限及其他角色菜单授权。
- 路径/方法白名单和按接口权限映射。
- 内部身份头清理、服务密钥错误和直接访问拒绝。
- 上游成功、业务错误、连接失败和超时响应。
- 旧登录和账号管理路由不再可达。

前端测试与浏览器验收覆盖：

- 三个菜单权限的显示、隐藏和路由访问。
- 消息筛选、分页、AI 总结、图片导出和数据库浏览。
- 401 退出、403 提示及网关不可用状态。
- 桌面与移动视口无溢出、遮挡，控制台无新增错误。
- TypeScript 检查和生产构建通过。

## 发布顺序与回滚

1. 部署支持可信网关鉴权的机器人后端。
2. 配置并部署兔兔窝后端网关及菜单权限。
3. 部署合并后的兔兔窝管理前端。
4. 验证主要流程后停止部署旧机器人前端，并发布 README 停用通知。

回滚时先恢复旧机器人前端部署。旧账号数据和旧鉴权实现仍保留，因此无需恢复数据库；重新启用旧路由需要回滚机器人后端版本。
