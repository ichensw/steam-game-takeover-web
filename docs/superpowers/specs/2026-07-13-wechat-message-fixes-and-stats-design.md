# 微信消息修复与聊天统计设计

## 目标

修复微信消息查询页的时间展示和消息类型文案，并在兔兔窝接龙后台增加一个受统一管理员权限控制的聊天统计页面。

## 已确认范围

- 消息列表展示机器人接口返回的 `createdAt.text`。
- 消息类型 `10002` 展示为“拍一拍”，并可作为筛选项。
- 聊天统计默认查询最近 7 个自然日，默认统计全部群聊，可选择单个群聊。
- 指标包含总消息数、活跃聊天人数、活跃群聊数、人均消息数、每日趋势和聊天人员排行。
- 统计数据由机器人后端直接通过 SQL 聚合；前端不跨分页下载消息做计算。

## 接口设计

机器人后端新增 `GET /api/stats/daily`，继续由 `trustedAdmin` 中间件保护。主后台通过现有 `/api/admin/wechat-bot/*path` 网关转发，并保留管理员 Bearer Token 鉴权、管理员身份请求头和服务密钥校验。

查询参数：

- `start`: 必填，`YYYY-MM-DD`，按机器人配置时区解释，包含该日 00:00:00。
- `end`: 必填，`YYYY-MM-DD`，包含该日，SQL 上界使用下一日 00:00:00。
- `roomId`: 可选；空值表示全部群聊。
- 日期范围最多 90 天，`start` 不得晚于 `end`。

响应 `data`：

```json
{
  "range": { "start": "2026-07-07", "end": "2026-07-13" },
  "roomId": "",
  "totals": {
    "messageCount": 1280,
    "participantCount": 86,
    "groupCount": 12,
    "messagesPerParticipant": 14.88
  },
  "daily": [
    { "date": "2026-07-07", "messageCount": 180, "participantCount": 31, "groupCount": 8 }
  ],
  "participants": [
    { "senderWxid": "wxid_x", "senderName": "用户", "messageCount": 72, "activeDays": 6, "groupCount": 3 }
  ]
}
```

`daily` 必须补齐范围内没有消息的日期并填 0。人员排行按消息数倒序，最多返回 100 人；人员唯一性以非空 `sender_wxid` 为准，显示名称取该用户最新一条非空 `sender_name`，无名称时回退 wxid。无有效发送人标识的记录计入消息数和群聊数，但不计入活跃人数和排行。人均消息数为总消息数除以活跃人数，无活跃人数时为 0，保留两位小数。

## SQL 聚合

所有查询都使用 `created_at >= ? AND created_at < ?`，可选追加 `room_id = ?`。分别执行：

- 总计：`COUNT(*)`、`COUNT(DISTINCT NULLIF(sender_wxid, ''))`、`COUNT(DISTINCT room_id)`。
- 每日趋势：按配置时区把 Unix 秒转换为日期后分组，并统计消息、人员和群聊去重数。
- 人员排行：按 `sender_wxid` 分组，统计消息数、活跃日期数和群聊数，限制 100 条。

暂不增加每日汇总表。当前数据量和最多 90 天范围适合即时聚合，避免引入回填、补偿和一致性维护成本。若生产查询出现可测量的性能问题，再基于执行计划增加联合索引或预聚合表。

## 前端设计

新增 `/wechat-stats` 页面和“聊天统计”菜单项。页面沿用现有管理后台的 PageHeader、筛选卡片和表格样式：

- 顶部筛选：群聊下拉、日期范围（默认最近 7 天）、查询和重置。
- 四个紧凑指标块：总消息数、活跃人数、活跃群聊、人均消息数。
- 每日趋势用扫描友好的表格展示日期、消息数、活跃人数和活跃群聊，不额外引入图表依赖。
- 人员排行表展示排名、昵称/wxid、消息数、活跃天数和参与群聊数。
- 加载、空数据和请求失败沿用现有 Ant Design 行为。

微信消息时间在 API 类型中声明为 `{ unix: number; text: string }`，通过纯函数格式化，兼容异常或旧字符串值。消息类型映射也抽成可测试纯函数，包含 `10002 -> 拍一拍`。

## 权限

新增菜单权限键 `wechat-stats`：

- 纳入主后台全部合法菜单键和角色菜单配置列表。
- 超级管理员强制拥有，兼容数据库中已保存的旧菜单配置。
- 普通管理员和 Kook 管理员不默认拥有，可由超级管理员通过现有角色菜单页面授权。

## 错误与安全

- 参数错误返回 `400 PARAM_INVALID`，数据库错误返回 `500 QUERY_FAILED`，不返回 SQL 或内部错误细节。
- 新接口不接受浏览器直连；机器人后端仍只信任正确服务密钥和完整管理员身份请求头。
- 主后台网关只允许既定方法和路径，统计接口只允许 GET。
- 所有 SQL 值使用参数绑定，`roomId` 不拼接到 SQL。

## 测试与发布

- 前端单元测试覆盖时间文本、旧字符串兼容、拍一拍映射、菜单权限和路由。
- 机器人后端使用 `sqlmock` 覆盖默认 7 天以外的显式范围、单群筛选、零值日期补齐、排行与参数错误，并验证可信网关保护。
- 主后台后端覆盖代理路径/方法白名单和 `wechat-stats` 权限回填。
- 发布前运行两个 Go 仓库的 `go test -count=1 ./...`、前端 `npm test` 和 `npm run build`。
- 部署仅替换三个现有服务的构建产物并重启对应 systemd 服务；路由仍在现有域名和 `/api` 下，预期无需修改 Nginx。部署前保留备份，健康检查与生产接口验证失败则恢复备份。

## 方案取舍

- 采用后端 SQL 聚合：数据完整、网络传输小、权限边界清晰。
- 不采用浏览器分页聚合：会漏掉未加载分页，性能和准确性都不可控。
- 暂缓预计算汇总表：当前需求没有证明需要额外的数据管道与一致性成本。
